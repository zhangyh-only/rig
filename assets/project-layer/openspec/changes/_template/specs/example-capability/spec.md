<!--
spec-delta 模板（本次变更对某一能力的增量）。
把目录名 <capability> 替换成具体能力域（如 user-auth、order-export），与 openspec/specs/ 下的能力域一一对应；同一 change 涉及多个能力域就建多个 specs/<capability>/spec.md。

写法：
- 只写"本次变更"对该能力的增量，不是能力全文。
- 三段按需保留——没有就整段删掉，别留空段。
- 每条需求一句"系统应当…"，紧跟一行"验收："给出可执行断言/测试（命令、用例名或可判定条件），对接 L0 自验证与 L4 完成度统计。
- MODIFIED 写改成什么样的新行为（覆盖旧描述，不写 diff）；REMOVED 写删掉什么及原因。

归档：tasks 全勾选后跑 `openspec archive <change>`，本 delta 会合并进 openspec/specs/<capability>/spec.md，成为该能力的当前权威。
填写时删掉所有 <尖括号占位>、示例条目和本注释块。
-->

# <capability> 能力规格增量

## ADDED Requirements

### 需求：<新增能力的一句话名称>

系统应当 <用一句话描述新增的行为：在什么条件下，对谁，做到什么可观测的结果>。

验收：<可执行断言/测试，如 `pytest tests/<area>_test.py::test_<case>`，或"调用 X 返回 Y / 状态变为 Z"等可判定条件>

### 需求：<第二条新增能力>

系统应当 <可观测行为>。

验收：<对应测试或断言>

## MODIFIED Requirements

> 改的是已存在于 openspec/specs/<capability>/ 的需求。直接写改后的完整新行为（覆盖旧描述），并保持需求名与原条目一致以便归档时定位替换。

### 需求：<被修改需求的原名称>

系统应当 <改后应当满足的新行为，把变化点说全，旧的怎样不必赘述>。

验收：<能证明新行为生效、且旧行为不再出现的测试或断言>

## REMOVED Requirements

> 仅当本次明确移除某条已有需求时保留本段，需求名与被删条目一致。

### 需求：<被移除需求的原名称>

移除原因：<为什么不再需要：能力下线 / 被新需求取代（指明取代它的需求）/ 合并进他处>。

验收：<可判定该行为确已消失，如对应用例已删除且 `grep -r <旧入口>` 无残留 / 调用旧接口返回 404>
