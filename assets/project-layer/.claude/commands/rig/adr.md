为本次决策创建一条新的 ADR（Architecture Decision Record）。决策标题：$ARGUMENTS

ADR 是跨域决策"为什么这么定"的**唯一权威**。feature-spec、AGENTS.md、代码注释等只**链接** ADR，**不复制**其内容。若标题为空，先问用户这条决策要记什么，再继续。

## 路由边界
- 适用于 Graph 编排边界、跨域技术选型、难回退架构选择等需要长期保留原因的决策。
- 不用于普通实现说明、临时任务记录或代码现状反扫。

执行步骤：

1. 读模板 @docs/adr/0000-template.md。若不存在，提示用户该项目尚未初始化 ADR 体系（缺模板），询问是否需要先补上模板，不要凭空生造格式。

2. 确定下一个编号：扫描 `docs/adr/` 下形如 `NNNN-*.md` 的文件（忽略 `0000-template.md` 和 `README.md`），取最大编号 +1，左补零到 4 位（如已有 0007，则新编号为 `0008`）。若目录里只有模板，则从 `0001` 起。

3. 由 $ARGUMENTS 生成 slug：转小写、空格与标点替换为连字符、去掉首尾连字符、中文标题可保留中文或按用户给的英文短语取 slug。最终文件名为 `docs/adr/NNNN-<slug>.md`。

4. 以模板为骨架创建该文件，并填好这些已知字段：
   - 标题用 $ARGUMENTS。
   - 编号 = NNNN。
   - 状态 = `Proposed`（提议中）。
   - 日期 = 今天。
   - 其余正文（背景 / 决策 / 考虑过的方案 / 后果）保留模板的 `<占位>` 提示，由用户填写；不要替用户编造理由或方案。

5. 更新 `docs/adr/README.md` 索引：在 ADR 列表中追加一行，链接到新文件，列出编号、标题、状态、日期。保持现有表格/列表格式一致；若 README 不存在则按模板风格新建一个最简索引（一行说明 + 列表/表格）。

6. 完成后向用户汇报：新文件路径、分配的编号，并提示——
   - 把决策的"为什么"写在这条 ADR 里，这是唯一权威；
   - 其它文档（feature-spec §决策链接区、AGENTS.md 等）只贴本 ADR 的链接，不要复制内容；
   - 决策落定后把状态从 `Proposed` 改为 `Accepted`；被取代时改为 `Superseded` 并链接到新 ADR。

注意：只新建/更新这两处文件（新 ADR + README 索引），不改动其它文件，不执行 git 提交。
