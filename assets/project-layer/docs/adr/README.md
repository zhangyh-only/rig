# 架构决策记录（ADR）

> ADR 是本项目**跨域决策"为什么"的唯一权威**。一旦某个取舍写进 ADR，其它文档（feature-spec、conventions、代码注释）只**链接**对应编号，不复制理由——避免多份 why 漂移。
> 它记的是"为什么这么定"，不是"现在怎么搭的"（后者归 feature-spec）。

## 编号规则
- `0000-template.md` 是模板，照抄改填，不参与计数。
- 正式 ADR 从 `0001` 起，四位数字递增，文件名 `NNNN-<短横线标题>.md`（如 `0001-用-postgres-而非-mongo.md`）。
- 编号一经分配**不复用、不重排**；废弃的决策保留文件、状态改 `Superseded`，并在新 ADR 里反向链接旧编号。

## 何时该写 ADR
满足任一条就值得写：
- **跨域 / 影响面大**：决策约束多个模块或团队，不是某个文件的局部实现。
- **难回退**：选型、数据模型、对外契约、技术栈等，改回来代价高。
- **选了 A 没选 B 的取舍**：存在被认真考虑过又被否决的备选项，理由值得留档，免得日后有人重复纠结。

## 何时不写
- 局部实现细节、可随手重构的写法 —— 归 code review / conventions。
- 只影响单文件、回退几乎零成本的改动 —— 照常 plan + git 即可。
- 变更**进行中**的 why —— 先写在 openspec 提案里；归档时若属跨域长期决策，再"毕业"成一条 ADR。

## 状态取值
`Proposed`（提案中）→ `Accepted`（已采纳）→ `Superseded by NNNN`（被某新决策取代）/ `Deprecated`（已废弃，无替代）。
